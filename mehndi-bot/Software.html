SOFTWARE

Hand Sizing

The Mehndi Bot has the capability to recognize the size and shape of a user’s hand as it is placed on the center of the gantry system and resize the user’s chosen design to fit in the palm of their hand.  This process is done using Google’s open-source framework called MediaPipe which uses machine learning solutions to locate and mark “landmarks” on a given hand.  This tool allows us to extract the coordinates for the bounds of the user’s palm which are used to find the hand shape and size.  With these dimensions, the chosen design image is rescaled to fit within the bounds.

Image to G-code

Once we have the design in the right size, the next step is to convert the image from a jpg or jpeg format into G-code which is much easier to use to determine the necessary motor commands to make the gantry system follow the path of the design.  There are several steps to this process to minimize the loss of detail in the final G-code.  The first step was to convert the image from an image file (.jpg, .jpeg) into a bitmap (.bmp).  In the form that we use, the bitmap images are in black and white and are of higher quality that the compressed image formats (jpg and jpeg).  With this bitmap, we then used an open-source software tool called Potrace which converts bitmapped images into vector graphics, saved as a Scalable Vector Graphics (.svg) file.  Finally, we use the open-source svg-to-gcode python library in order to convert this vectorized svg into gcode commands.

Serial Communication

Once we have the G-code file, we need to send these commands over to the Arduino so they can be executed by the MehndiBot. In order to do this, we created a Python script to read each of the G-code lines and relay them across Serial to the Arduino.
 
First, we convert the G-code file into an array of commands. Then, we send one command after another through Serial. The program waits until the MehndiBot finishes running the command before sending the next one over. 

G-code Interpretation - Firmware

The final step in running a G-code file is converting the commands to motor movements and executing them. Having sent the commands over Serial already, we read each line from the Serial through an Arduino script and parse through the command to identify whether it is a G or M command. For our application, G1 commands lead to a linear movement from the current position to the new position while M commands lead to Z-directional movement. M3 commands move the syringe downwards and M5 commands move it upwards. 

Our G-code interpreter keeps track of the current position of the syringe (initiating (0, 0) at the starting point), and moves the device according to absolute positions. Once the x and y distance needed to travel has been determined, Breseham’s Line Algorithm is implemented to create a line with the slope required to reach from the current position to the new position. Finally, the current position is updated and the command that has just been executed is printed to the Serial to double check that no data has been lost.

Challenges in Development

The main challenge we faced during the process of writing our software was data loss across Serial. The first problem we had was that the Serial would skip reading some of the characters we were trying to send over. After troubleshooting step by step, we realized that we needed to match the baud rate that we were writing and reading with. 

Our program originally wrote an array of G-code commands over Serial step by step without waiting. The problem ended up being that it would send commands while the motors were still moving, which meant that the Arduino would miss reading these commands. This led to data loss and thus an incomplete image being drawn. To solve this, we currently have added a checkpoint through the Arduino code. Everytime the motors successfully read a command, Arduino prints that command back over Serial. Our Python script waits to check if a command has been relayed before sending over the next command. This checkpoint and the following delay if the motors are still running ensure that no data is lost during this communication. This was manually verified during testing by looking at the commands sent vs received by Serial and ensuring that none are missing.

Finally, the initial method we used to find the bounding box for the hand was inaccurate. We originally turned the image into a bitmap and found the contour points of the hand. The problem here was that the algorithm would either include background contours or miss details of the hands. In order to solve this, we currently have implemented a library from Google called MediaPipe that finds landmarks on the hand. This method was much more precise and improved the accuracy of our MehndiBot.
